# English Benz — Setup

The app is a single static HTML page. Progress and the exercise content repository
live in **Supabase** (a free Postgres database). Sign-in is by **email + password**,
so your history syncs across every device. Fresh exercises are generated by a small
**local script** you run when you want more content.

```
Browser app  ──reads exercises, writes progress──▶  Supabase (Postgres + Auth)
                                                        ▲
scripts/generate-content.mjs ──writes exercises──────────┘   (uses your Claude API key,
   (runs on your machine)                                     which never leaves it)
```

---

## 1. Create the two free accounts (you)

1. **GitHub** — https://github.com/signup (for hosting + backup).
2. **Supabase** — https://supabase.com → *Start your project* → sign in with GitHub.
   Then **New project**: pick a name, a strong database password, a region near you.
   Wait ~2 minutes for it to provision.

> Free-tier note: a Supabase project **pauses after ~7 days of inactivity**. If the
> app can't load exercises, open the Supabase dashboard and click *Resume*.

## 2. Create the database tables

Supabase dashboard → **SQL Editor** → **New query** → paste the whole of
[`db/schema.sql`](db/schema.sql) → **Run**. It's safe to re-run.

## 3. Turn on email/password sign-in

Dashboard → **Authentication → Providers → Email**: make sure **Email** is enabled.
For a personal app you can also turn **off** "Confirm email" (Authentication →
Providers → Email → *Confirm email*) so you can log in immediately without the
confirmation round-trip.

## 4. Get your keys

Dashboard → **Project Settings → API**. You need:

| Value | Used by | Secret? |
|---|---|---|
| **Project URL** | app + pipeline | no |
| **anon public** key | the browser app | no (safe to ship — row-level security protects data) |
| **service_role** key | the pipeline only | **YES — machine-only, never in the app or git** |

## 5. Fill the content repository (the local pipeline)

```bash
cd scripts
npm install
cp .env.example .env
```

Edit `scripts/.env` with your `ANTHROPIC_API_KEY` (from platform.claude.com), the
Supabase **Project URL**, and the **service_role** key. Then:

```bash
npm run generate            # ~8 exercises per level/type
node generate-content.mjs 12   # or a bigger batch
```

Re-run it any time you want more content — duplicates are skipped automatically.
This is the only paid step; the browser app itself makes no Claude API calls.

Each run stores the sentences it found in `source_documents` before generating
anything, and every exercise it writes points back at that row. Nothing is ever
generated from text that wasn't kept.

## 5b. Admin panel (that one account only)

Signed in as the admin address, the menu shows **⚙ Admin**. Four ways in, all of
which store their raw text and then produce translation, listening **and**
phrasal-verb exercises:

| Control | What it does |
|---|---|
| **Fetch fresh news** | Web-searches recent articles and builds exercises from real sentences |
| **Generate from URL** | Fetches a page server-side, strips the HTML, generates from the text |
| **Generate from PDF** | Extracts text in the browser, generates from it |
| **Generate from saved text** | Re-uses anything ingested earlier — no re-upload, no second search |

The Anthropic key lives only in the Edge Function's secrets, never in the page.

## 6. The app is wired to Supabase

`index.html` already has your Project URL and anon key in the `EB_CONFIG`
block near the top of its `<script>`. On load it shows a **Sign in / Create account**
screen; once signed in, all four games run, exercises come from the database
(least-seen-first per user), "times seen" is tracked per exercise, and every stat
syncs across your devices. With no connection it falls back to the built-in
offline packs.

> If you edit `db/schema.sql` (it gained a `kv` table for stats sync), re-run the
> whole file in the SQL editor — it's safe to re-run.

**Test it:** open `index.html` in a browser, create an account, and play a
round. Then open it in a different browser / device, sign in with the same account,
and confirm your stats are there.

## 7. Publish (GitHub Pages)

Push the repo to GitHub and enable **Pages** for a public URL — details when we get
there. The anon key in the file is safe to publish; row-level security protects the
data.
